## Task Tracker API
Внутренний сервис для управления задачами команды разработки. Реализован как отказоустойчивое асинхронное приложение с контейнеризацией и кэшированием.


## Быстрый запуск
Для запуска и поднятия всего окружения достаточно одной команды:

```bash
docker-compose up --build
```

После запуска сервис будет доступен по адресу: http://localhost:8000/docs или же http://127.0.0.1:8000/docs

## Принятые решения
 В модели добавлено `updated_at`, потому что это позволяет отслеживать активность по задаче без необходимости копаться в логах базы.

 Статусы (Enum). Для ограничение списка статусов на уровне БД и Pydantic для предотвращения некорректного ввода.

 Redis TTL 60 секунд. TTL выбрано 60 секунд, т.к. это оптимальное время, чтобы не гонять запросы в Postgres при каждом обновлении страницы командой, но при этом данные остаются свежими. Также при создании, обновлении или удалении задачи кэш в Redis полностью очищается, чтобы команда сразу видела актуальные изменения.

 Lifespan-контекст для корректного закрытия соединений с БД при остановке сервиса.

 Логика разделена на слои: `models`, `schemas`, `api/routes`, `db`.

 Учтена задача "Переход назад из `done` невозможен - возвращай понятную ошибку". Если задача переведена в статус `done`, её нельзя вернуть назад. При попытке такого действия API возвращает статус `400 Bad Request` с понятным описанием причины.


## Тестирование
Для запуска тестов внутри контейнера:

```bash
docker exec -it task_tracker_app pytest
```

Тесты покрывают создание задачи, успешную смену статуса, блокировку изменения статуса из состояния `done`.


## CI/CD
В репозитории настроен CI-пайплайн(GitHub Actions), который автоматически запускает тесты при каждом пуше в ветку `main`
